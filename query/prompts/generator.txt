## KQL Syntax Rules

**Boolean operators:**
- Use AND, OR, NOT (case-insensitive) to combine conditions
- AND has higher precedence than OR — use parentheses to override

**Term queries (exact match):**
- field: "value"
- Example: service.name: "payments"

**Range queries:**
- field >= value  /  field <= value  /  field > value  /  field < value
- For date math use: @timestamp >= now-1h  (no quotes needed for date math)
- For absolute dates use quotes: @timestamp >= "2026-04-01T00:00:00Z"
- Combined range: http.status_code >= 400 AND http.status_code < 500

**Term sets (multiple values for one field):**
- field: ("value1" OR "value2")
- Example: log.level: ("ERROR" OR "WARN")

**Wildcards and existence:**
- field: *   → field exists
- field: val*  → field starts with val

**Nested fields:**
- IMPORTANT: Fields marked as NESTED must use curly brace syntax
- items: { product_name: "laptop" AND quantity > 2 }

**Time ranges:**
- Always use @timestamp unless the schema specifies a different time field
- Date math reference: now-1h (1 hour ago), now-30m (30 minutes), now/d (start of today)

## Hard Rules
- ONLY use field names listed in the schema below — never invent field names
- NEVER output explanations, comments, or markdown — KQL query only
- NEVER use pipe syntax (no | where) — KQL uses AND for all conditions
- For nested fields annotated with "NESTED FIELD", always use curly brace wrapper
